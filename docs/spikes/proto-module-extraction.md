# Spike: extract the gRPC contract into a dedicated proto module/repo

Status: spike (design) · Execute at the SEAM (after the handle 0.16 wave settles) · Motivation: the contract is entangled with heddle's crate graph.

## Why now
The gRPC contract has become a first-class shared artifact (heddle CLI, weft server, tapestry web all consume it), but it lives inside heddle and is coupled to heddle's release cadence + crate graph. Concrete pains hit this session:
- **Version entanglement (the tell):** `weftctl` is stuck on `heddle-grpc` **0.13** because `heddle-client` transitively pins `^0.13`, so weft-server (0.15) and weftctl (0.13) run different contract versions. The proto being a crate *in* the heddle workspace is the root.
- **Manual npm dispatch** (`publish-npm-client.yml`) — a recurring footgun on every bump (0.14/0.15/0.16).
- **Additive-only by hand** — we eyeball wire-compat on every bump; nothing enforces it.
- **Ownership smell** — the proto describes the *hosted (weft)* surface but lives in *heddle*.

## Current state (grounded)
- Proto: `heddle/crates/grpc/proto/heddle/v1/*.proto` (~18 files: auth, common, content, feed, hosted, search, timeline, sync, tree_edit, transactions, signals, review, policies, repo_events, discussion, hooks, operations, service).
- Rust SDK: `heddle-grpc` crate, generated in `crates/grpc/build.rs` (tonic/prost + a `heddle_descriptor.bin`).
- TS SDK: `@heddleco/grpc`, generated with `@bufbuild/protoc-gen-es` (`clients/grpc`), regenerated by `heddle-devtools grpc-ts`.
- In-tree consumers use **path deps**: `grpc = { path = "../grpc", package = "heddle-grpc" }` in `crates/{client,cli,daemon}`.
- Out-of-tree consumers use **published** packages: weft ← `heddle-grpc` (crates.io), tapestry ← `@heddleco/grpc` (GitHub Packages).

## Decisions

### 1. Repo — DECISION: a dedicated `heddleco/api` repo
Its only job: the `.proto` source + buf config + codegen + the SDK publish pipeline. Name `api` over `proto` (it's the API contract; SDKs generate *from* it). Neutral owner of the contract, decoupled from any service.

### 2. Tooling — DECISION: adopt **buf**
- `buf.yaml` + `buf.gen.yaml`. `buf lint` (style), **`buf breaking`** against the published module/`main` (AUTOMATES the additive-only guarantee we hand-check), `buf generate` for every SDK.
- TS: keep `protoc-gen-es` (already the toolchain). Rust: generate via a buf plugin (`protoc-gen-prost`/`protoc-gen-tonic`) OR keep `tonic-build` driven from the module — **parity-check required** (§Risks). Retires `heddle-devtools grpc-ts` + the ad-hoc checks.

### 3. Publish — DECISION: automated multi-SDK publish on tag
A version tag in `heddleco/api` → CI generates + publishes `heddle-grpc` (crates.io) + `@heddleco/grpc` (npm) **automatically** — no manual dispatch. Adding a Python/Go SDK later is a `buf.gen.yaml` entry + a publish step.

### 4. Versioning — DECISION: continue the line, don't reset
The api repo continues `heddle-grpc`/`@heddleco/grpc` from **0.16 → 0.17** (not back to 0.1). The first api-repo release (0.17) should be a **no-op content bump** — byte-identical generated output to 0.16 — to prove parity before any consumer cuts over.

### 5. In-tree heddle consumers — DECISION: consume the published crate (like weft)
`heddle/crates/{client,cli,daemon}` switch from `path = "../grpc"` to the **published `heddle-grpc`**. Heddle then consumes its own contract as a normal dep — same as weft/tapestry — which is what makes the contract independent. (Rejected: git-submodule/vendored path — keeps the coupling.)

### 6. The `weftctl@0.13` knot — DECISION: audit + break during migration
Extraction alone doesn't move the pin (`heddle-client ^0.13`). But (a) once `heddle-grpc` publishes independently, `heddle-client` bumps its pin in a fast api-only release, and (b) **audit whether weftctl needs `heddle-client` at all or just `heddle-grpc`** — if the latter, depend on `heddle-grpc` directly and the transitive knot dissolves. Resolve this as part of Phase 1.

## Migration plan (executable, phased)
- **Phase 0 — stand up `heddleco/api`:** move the `.proto` + add buf config + port the Rust+TS codegen + the publish CI. Publish **0.17** from there — verified byte-identical to 0.16 (parity gate). No consumer changes yet.
- **Phase 1 — cut consumers over:** re-point heddle `{client,cli,daemon}` path-deps → published `heddle-grpc` 0.17; weft + tapestry bump to 0.17 (from the new source, same registries). Do the weftctl/heddle-client audit (§6). Delete `heddle/crates/grpc`.
- **Phase 2 — turn on the guarantees:** enable `buf breaking` as a required CI gate + the automated tag→publish; retire `heddle-devtools grpc-ts` and `publish-npm-client.yml`'s manual dispatch.
- **Verify:** all three repos build + CI green on 0.17; a trial additive change flows api-repo → publish → consumer bump cleanly, and a trial breaking change is *caught* by `buf breaking`.

## Risks / costs (honest)
- **Codegen parity:** the Rust side (build.rs descriptor + tonic/prost) must reproduce byte-identical output via buf generate — the 0.17 parity gate exists to catch drift before cutover. If parity is hard, keep `tonic-build` invoked from the module rather than a buf plugin.
- **Publish→consume latency for heddle-internal proto changes:** today a heddle proto change is instant (path dep); after extraction it goes through an api-repo release. That's the *cost* of decoupling — acceptable given the benefits, but real for fast iteration; the automated tag→publish keeps it cheap.
- Another repo + CI + permissions to run.
- One-time migration effort (mechanical: move files, rewire ~6 pins, port 2 codegen paths, 1 publish pipeline).

## Open decisions the execution resolves
1. Repo name (`api` vs `proto`).
2. Rust codegen: buf plugin vs `tonic-build`-from-module (decided by the parity gate).
3. weftctl → `heddle-client` vs direct `heddle-grpc` (the §6 audit).
4. Where the descriptor/reflection bits live post-move.
